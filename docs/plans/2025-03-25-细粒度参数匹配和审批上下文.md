# 细粒度参数匹配 & 审批上下文显示

**日期:** 2025-03-25
**状态:** 已实现

---

## 1. 问题背景

### 问题 1：审批时缺少上下文

当前 `REQUIRES_APPROVAL` 返回示例：

```
[Approval required] ask before running commands
Request ID: a1b2c3d4-...
Reply APPROVE a1b2c3d4-... to allow, or DENY a1b2c3d4-... to block.
```

用户只能看到 "ask before running commands"，不知道代理实际要执行什么命令，无法做出知情决策。

### 问题 2：工具内参数级细分

当前 `exec` 作为单一工具整体设置权限，但：
- `ls`、`cat` 等读取类命令风险低
- `rm`、`edit` 等写操作风险高
- `sudo`、curl 下载执行等高危

用户希望在 `rules.json` 中按参数值再细分，而不是按工具名整体一刀切。

---

## 2. 设计方案

### 2.1 审批上下文显示

**修改文件:** `openclaw-plugin/src/index.ts`

在 `REQUIRES_APPROVAL` 分支中，把具体命令提取出来拼到 `blockReason` 里：

```typescript
// 从 toolCall.args 中提取命令（尝试多个常见字段名）
function extractCommand(args: Record<string, unknown>): string | null {
  for (const key of ['command', 'cmd', 'script', 'code', 'args']) {
    const val = args[key];
    if (typeof val === 'string' && val.length > 0) return val;
    if (Array.isArray(val)) return val.join(' ');
  }
  return null;
}

// 返回时包含命令上下文
if (result.decision === 'REQUIRES_APPROVAL') {
  const cmd = extractCommand(params);
  const cmdLine = cmd ? `\nCommand: ${cmd}` : '';
  const uuid = createApprovalRequest(toolName, params, result.reason ?? 'No matching rule');
  return {
    block: true,
    blockReason:
      `[Approval required] ${result.reason ?? 'No matching rule'}${cmdLine}\n\n` +
      `Request ID: ${uuid}\n\n` +
      `Reply APPROVE ${uuid} to allow, or DENY ${uuid} to block.`,
  };
}
```

效果：

```
[Approval required] ask before running commands
Command: rm -rf /tmp/some-files

Request ID: a1b2c3d4-...

Reply APPROVE a1b2c3d4-... to allow, or DENY a1b2c3d4-... to block.
```

### 2.2 `argsPattern` 参数级匹配

#### 2.2.1 类型定义

**修改文件:** `src/types.ts`

```typescript
export interface CompiledRule {
  action: RuleAction;
  tool?: string;
  toolPattern?: string;
  intentPattern?: string;
  /** Regex to match against tool call arguments (joined as space-separated string) */
  argsPattern?: string;
  reason: string;
}
```

#### 2.2.2 匹配逻辑

**修改文件:** `src/matcher.ts`

当前匹配逻辑回顾：
- 遍历 rules，第一个匹配决定结果
- `block` 优先级最高（遇到即停）
- `tool` 精确匹配或 `toolPattern` 正则匹配
- `intentPattern`（如有）匹配意图文本

新增：`argsPattern` 匹配

```typescript
function matchArgs(rule: CompiledRule, toolCall: ToolCall): boolean {
  if (!rule.argsPattern) return true; // 无 argsPattern 则跳过
  // 将所有参数值拼成字符串用于匹配
  const argsStr = Object.values(toolCall.args)
    .filter(v => typeof v === 'string' || typeof v === 'number')
    .join(' ');
  return new RegExp(rule.argsPattern).test(argsStr);
}
```

匹配流程改为：

```typescript
for (const rule of rules) {
  // 1. 工具名匹配
  const toolMatch =
    (rule.tool && rule.tool === toolCall.toolName) ||
    (rule.toolPattern && new RegExp(rule.toolPattern).test(toolCall.toolName));

  // 2. 参数匹配（如果规则指定了 argsPattern）
  const argsMatch = matchArgs(rule, toolCall);

  // 3. 意图匹配
  const intentMatch =
    !rule.intentPattern ||
    new RegExp(rule.intentPattern, 'i').test(intent.text);

  if (toolMatch && argsMatch && intentMatch) {
    // ... 现有优先级逻辑
  }
}
```

#### 2.2.3 规则示例

```yaml
# rules.yaml
- block exec when command matches "rm -rf" - never delete recursively
- require approval before exec when command matches "^(rm|mv|edit|tee)" - write operations need approval
- allow exec when command matches "^(ls|cat|head|tail|grep|pwd|which|file)" - read-only commands
```

编译后 `rules.json`：

```json
{
  "rules": [
    {
      "action": "block",
      "tool": "exec",
      "argsPattern": "rm -rf",
      "reason": "never delete recursively"
    },
    {
      "action": "require_approval",
      "tool": "exec",
      "argsPattern": "^(rm|mv|edit|tee)",
      "reason": "write operations need approval"
    },
    {
      "action": "allow",
      "tool": "exec",
      "argsPattern": "^(ls|cat|head|tail|grep|pwd|which|file)",
      "reason": "read-only commands"
    }
  ]
}
```

**优先级说明：** 第一条 `block rm -rf` 会优先匹配并阻止，满足问题中"高危直接阻止"的需求；一般 `rm`（非 -rf）会落入第二条需要审批。

#### 2.2.4 Middleware 层透传

**修改文件:** `src/middleware.ts`

`argsPattern` 直接存在 `CompiledRule` 里，`matcher.ts` 已处理，`middleware.ts` 无需修改。

## 4. 向后兼容

- `argsPattern` 是**可选字段**，无此字段的现有规则不受影响
- 匹配逻辑：当规则无 `argsPattern` 时，退化为现有行为
- YAML 语法不变（编译器会透传 `argsPattern` 由 LLM 推断生成）

---

## 5. 未解决 / 后续

- 编译器（`compiler.ts`）如何从自然语言推断 `argsPattern`？目前 LLM 可能难以稳定生成精确的正则。初期可让用户在 YAML 中用结构化语法标注，LLM 透传：

  ```yaml
  - allow exec argsPattern="^(ls|cat)" - read-only
  ```

  长期可微调 `COMPILER_PROMPT` 提升推断质量。

- 通用工具（`write`、`edit`）的 `argsPattern` 应用场景：按写入路径细分（如只允许写 `/tmp/`）
